     Plan: Build the GitHub Data Ingester

     Context

     The ingester.py file currently contains only a docstring. We need a working
     ingester that fetches issues and PRs from configured GitHub repos via the
     REST API, handles rate limits, and writes to a local SQLite database. This
     is the data foundation — the query engine and UI depend on it.

     Files to modify

     - schema.sql — Add full DDL (CREATE TABLE statements for repos and issues)
     - ingester.py — Full implementation

     Schema Design (schema.sql)

     CREATE TABLE IF NOT EXISTS repos (
         id INTEGER PRIMARY KEY,
         owner TEXT NOT NULL,
         name TEXT NOT NULL,
         full_name TEXT NOT NULL UNIQUE,
         description TEXT,
         stars INTEGER,
         forks INTEGER,
         open_issues_count INTEGER,
         language TEXT,
         created_at TEXT,
         updated_at TEXT
     );

     CREATE TABLE IF NOT EXISTS issues (
         id INTEGER PRIMARY KEY,
         repo_id INTEGER NOT NULL REFERENCES repos(id),
         number INTEGER NOT NULL,
         title TEXT NOT NULL,
         body TEXT,
         state TEXT NOT NULL,          -- 'open' or 'closed'
         is_pr INTEGER NOT NULL DEFAULT 0,
         author TEXT,
         labels TEXT,                  -- JSON array of label names
         created_at TEXT,
         updated_at TEXT,
         closed_at TEXT,
         url TEXT NOT NULL,            -- html_url for clickable evidence
         UNIQUE(repo_id, number)
     );

     Ingester Design (ingester.py)

     Public interface

     async def ingest_repos(repos: list[str], db_path: str, max_items: int = 
     200, on_progress=None) -> None

     - repos: list of "owner/repo" strings
     - db_path: path to SQLite file (default data/github.db)
     - max_items: max issues+PRs per repo (default 200 for demo speed)
     - on_progress: optional callback (message: str, pct: float) for UI progress

     Internal structure

     1. _init_db(db_path) — Read schema.sql, execute DDL, return connection
     2. _fetch_repo_meta(client, owner, repo) — GET /repos/{owner}/{repo} →
     insert into repos
     3. _fetch_issues(client, owner, repo, max_items) — Paginate GET 
     /repos/{owner}/{repo}/issues?state=all&per_page=100&sort=updated (returns
     both issues and PRs). Stop at max_items.
     4. _handle_rate_limit(response) — Check X-RateLimit-Remaining header; if 0,
     sleep until X-RateLimit-Reset. Also handle 403/429 with exponential
     backoff.
     5. _save_issues(conn, repo_id, items) — Batch INSERT OR REPLACE into issues
     table. Parse labels into JSON array of names. Set is_pr based on presence
     of pull_request key.

     Rate limit & retry strategy

     - Check X-RateLimit-Remaining before each page; if ≤1, sleep until reset
     time
     - On 403/429: exponential backoff (1s, 2s, 4s) up to 3 retries
     - On 5xx: retry up to 3 times with backoff
     - httpx.AsyncClient with 30s timeout

     CLI mode

     if __name__ == "__main__":
         asyncio.run(ingest_repos(DEFAULT_REPOS, DEFAULT_DB_PATH))

     Where DEFAULT_REPOS = ["tiangolo/fastapi", "pydantic/pydantic"] and
     DEFAULT_DB_PATH = "data/github.db".

     Key decisions

     - Async httpx for concurrent page fetches (one repo at a time to respect
     rate limits, but pages fetched sequentially per repo).
     - INSERT OR REPLACE so re-ingestion is idempotent.
     - body truncated to first 500 chars to keep DB small while preserving
     enough context for NL queries.
     - Labels stored as JSON string — simple for the LLM to query with
     json_each() or LIKE patterns.
     - data/ directory auto-created if missing.

     Verification

     1. cp .env.example .env and fill in a valid GITHUB_TOKEN
     2. python ingester.py — should create data/github.db with ~200 rows per
     repo
     3. Verify: sqlite3 data/github.db "SELECT count(*) FROM issues; SELECT 
     count(*) FROM repos;"
     4. Check rate-limit handling: inspect stdout logs during ingestion

     Plan: Build the NL→SQL Query Engine

     Context

     The query engine is the core intelligence layer. It takes a
     natural-language question, generates a safe SELECT query against the SQLite
     schema, executes it, and synthesises a grounded answer with citations. The
     LLM is accessed via an OpenAI-compatible proxy (custom base URL), so the
     client must be configured with base_url and api_key from environment
     variables.

     Files to modify

     - query_engine.py — Full implementation
     - .env.example — Add OPENAI_BASE_URL variable

     .env.example additions

     OPENAI_BASE_URL=https://your-proxy.example.com/v1
     OPENAI_API_KEY=sk-key_here
     OPENAI_MODEL=gpt-4o

     Query Engine Design (query_engine.py)

     Public interface

     async def ask(question: str, db_path: str = DEFAULT_DB_PATH) -> Answer

     Returns an Answer dataclass:
     @dataclass
     class Answer:
         question: str
         sql: str
         raw_rows: list[dict]
         text: str           # synthesised natural-language answer
         error: str | None   # set if something went wrong

     Two-call LLM flow

     1. SQL generation call: System prompt includes the full schema DDL +
     instructions to produce ONLY a single SELECT statement. User message is the
     question. Parse the SQL from the response.
     2. Validation: Check that the generated SQL starts with SELECT (strip
     comments/whitespace). Reject anything with
     INSERT/UPDATE/DELETE/DROP/ALTER/CREATE/ATTACH etc.
     3. Execution: Run against SQLite, fetch rows (cap at 50 rows to avoid
     blowing context).
     4. Synthesis call: Feed the question + SQL + result rows to the LLM. System
     prompt instructs it to answer grounded in the data, cite issue/PR numbers
     with URLs, and say "I don't have enough data" if results are empty.

     OpenAI client setup

     from openai import AsyncOpenAI

     client = AsyncOpenAI(
         base_url=os.getenv("OPENAI_BASE_URL"),  # required — custom proxy
         api_key=os.getenv("OPENAI_API_KEY"),
     )
     MODEL = os.getenv("OPENAI_MODEL", "gpt-4o")

     SQL safety validation

     A simple allowlist approach:
     - Strip leading whitespace/comments
     - First keyword must be SELECT (case-insensitive)
     - Reject if any of these keywords appear as standalone words: INSERT,
     UPDATE, DELETE, DROP, ALTER, CREATE, ATTACH, DETACH, PRAGMA
     - Execute with conn.execute() (not executescript) so only one statement
     runs

     System prompts

     SQL generation prompt (key points):
     - Include full schema DDL inline
